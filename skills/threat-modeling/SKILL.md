# Threat modeling

## When to use

- First step the coordinator runs, before any persona is dispatched
- When a codebase has materially changed (new payment provider, added multi-tenancy, introduced AI features)
- When a human reviewer wants to sanity-check which personas are in scope

## Why STRIDE-lite

Classic STRIDE (Spoofing / Tampering / Repudiation / Information disclosure / Denial of service / Elevation of privilege) is useful but heavy. For a 15-minute pre-flight the coordinator only needs enough information to:

1. Decide which personas to spawn.
2. Decide which adapter overlay to load.
3. Set data-sensitivity context for severity scoring.

That is a small fixed set of facts, not a freeform threat narrative. The output is a one-page declaration that every downstream persona reads once.

## Inputs

The coordinator gathers these by reading the repo — no interview required.

- `package.json`, `pyproject.toml`, `go.mod`, etc. for stack detection
- `README.md` and any `docs/` for stated purpose
- `.env.example` / `.env.sample` for provider hints (STRIPE_, OPENAI_, CLERK_, AUTH0_, SUPABASE_)
- `supabase/`, `prisma/`, `drizzle/`, `db/migrations/` for schema clues (PII columns, `org_id`, `tenant_id`)
- Infra markers: `Dockerfile`, `vercel.json`, `wrangler.toml`, `fly.toml`, `netlify.toml`, `.github/workflows/`

## Output: THREAT_MODEL.md

The coordinator writes exactly one file, `docs/red-team-<date>/THREAT_MODEL.md`, using the template below. Every field is required; use `none` or `unknown` rather than omitting.

### Template

```markdown
# Threat model — <project>

_Generated by the red-team coordinator on <date>._

## 1. Purpose and users

- **What the app does** (one sentence, from README):
- **Primary user class** (consumer / SMB / enterprise / internal / developer):
- **Anonymous use permitted** (yes / no / partial — describe):

## 2. Data classification

Tick all that apply. Drives severity ceilings for every persona.

- [ ] **PII** — names, emails, phone, addresses
- [ ] **PHI** — health/medical data (HIPAA scope)
- [ ] **PCI** — cardholder data handled in-app (not just tokenised)
- [ ] **Financial non-PCI** — balances, transactions, invoices
- [ ] **Credentials** — stored passwords, API keys for third parties, OAuth tokens
- [ ] **User-generated content** — uploads, documents, messages
- [ ] **None of the above** — low-stakes data only

Notes (schema tables/columns that justify each tick):

## 3. Multi-tenancy model

- [ ] **Single-tenant** — one org per install, no cross-user isolation needed
- [ ] **Personal accounts only** — each user is their own tenant
- [ ] **Org-based** — users belong to one org at a time
- [ ] **Org-based with multi-membership** — users belong to multiple orgs
- [ ] **Hierarchical** — orgs contain sub-orgs / teams / projects

Tenancy key(s) in schema (`org_id`, `tenant_id`, `account_id`, etc.):
Primary enforcement layer (`RLS` / `application-layer assertRole` / `both` / `none`):

## 4. Payment provider

- [ ] **None — no money changes hands**
- [ ] **Stripe** (subscriptions / one-off / Connect / Checkout — tick which)
- [ ] **Paddle / Lemon Squeezy / Chargebee** (merchant-of-record)
- [ ] **PayPal**
- [ ] **Other:** ____

Webhook endpoints present: yes / no. Paths:

## 5. AI features

- [ ] **None**
- [ ] **LLM text generation** — provider: ____
- [ ] **LLM with user-supplied prompts passed to model**
- [ ] **Agentic / tool-calling** — tools exposed:
- [ ] **Vector search / RAG** — store:
- [ ] **Fine-tuning on user data** — destination:

Per-request cost ceiling enforced (yes / no / unknown):

## 6. Auth provider

- [ ] **Own** — password + session cookies
- [ ] **Clerk**
- [ ] **Auth0**
- [ ] **Supabase Auth**
- [ ] **NextAuth / Auth.js**
- [ ] **WorkOS / Cognito / Firebase Auth / other:** ____
- [ ] **Magic link only**
- [ ] **OAuth-only (no passwords)**

MFA available (yes / no / optional):

## 7. Infra provider

- [ ] **Vercel**
- [ ] **Cloudflare Workers / Pages**
- [ ] **Netlify**
- [ ] **AWS** (ECS / Lambda / EC2 / App Runner — tick)
- [ ] **GCP** (Cloud Run / GKE / other)
- [ ] **Fly.io / Render / Railway**
- [ ] **Self-hosted / on-prem**

Database: ____
Object storage: ____
Background jobs: ____ (QStash / Inngest / SQS / cron / none)

## 8. Persona selection (filled by coordinator)

Based on the sections above, the coordinator will spawn:

- [ ] external-attacker (always)
- [ ] authz-reviewer (always)
- [ ] tenancy-reviewer (if 3 is org-based or hierarchical)
- [ ] payment-reviewer (if 4 is not "none")
- [ ] ai-reviewer (if 5 is not "none")
- [ ] supply-chain-reviewer (always)
- [ ] crypto-reviewer (if credentials stored or webhooks present)
- [ ] webhook-reviewer (if 4 or 5 uses webhooks)
- [ ] infra-reviewer (always)
- [ ] insider-threat-reviewer (if staff roles exist or >10 FTE in org)
- [ ] denial-of-wallet-reviewer (if 5 is LLM or section 4 metered)
- [ ] data-exfil-reviewer (if 2 includes PII/PHI/PCI/credentials)

## 9. Adapter detection (filled by coordinator)

Detected stack adapter: ____ (one of `next-app-router`, `next-pages-router`, `sveltekit`, `remix`, `express`, `fastify`, `fastapi`, or `none`)
```

## How personas consume this

Each persona opens `THREAT_MODEL.md` at start of run and:

- Skips itself if not on the selection list in section 8
- Uses section 2 to set its impact ceiling (PHI present means any info-disclosure starts at Medium Impact)
- Uses section 3 to know whether tenancy bugs matter
- Uses section 6 to know which auth primitives to check

## Anti-patterns

- Filling fields from the README when the schema contradicts it. Code is truth.
- Skipping personas because "the app doesn't handle money" without confirming section 4.
- Leaving a field blank rather than writing `none` or `unknown` — downstream personas then silently skip checks.
- Re-running the threat model for every persona. It runs once per review.
